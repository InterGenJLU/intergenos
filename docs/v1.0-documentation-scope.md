# InterGenOS v1.0 Documentation Scope

**Last updated:** 2026-04-30
**Status:** Proposal — pending owner sign-off
**Purpose:** Inventory current documentation, identify v1.0-must-ship gaps, and recommend a writing-and-deferral plan.

This document is the working scope for what InterGenOS publishes alongside the v1.0 release. The audience is the maintainers planning the launch and the future contributors who will inherit the documentation tree. It is not itself end-user-facing.

InterGenOS is a small project. The documentation tree should be small too — only what users, contributors, and reviewers actually need. Every document here either earns its place or moves to a v1.0+1 follow-up.

## 2. v1.0-must-ship gaps

These are documents InterGenOS should not launch v1.0 without. Each entry includes the proposed filename, a one-paragraph scope, and the dependency that gates the writing.

### 2.1 `docs/getting-started.md` — first-time user orientation

**Audience:** A person who has just downloaded the v1.0 ISO and wants to know what to do with it.
**Scope:** Hardware requirements, image verification (sha256 + PGP signature against `signing-key.md`), USB write commands, what to expect at first boot (link to `first-boot-greeter.md`), what to expect at first login, where to go next. Five to eight minutes of reading.
**Dependency:** None — can be written today.

### 2.2 `docs/install-guide.md` — Forge installer walkthrough

**Audience:** A person who has booted the v1.0 ISO and is staring at the Forge TUI.
**Scope:** Step-by-step through the Forge install flow — partitioning choices, hostname/locale, account creation, Secure Boot MOK enrollment, bootloader stage, post-install hooks. Includes "what to do if `efibootmgr` fails" and "what to do if MOK enrollment is rejected." Screenshots when the TUI is stable enough to capture.
**Dependency:** Forge installer must reach a UI-stable state. Currently in active development; see `installer/` and the audit lane (DS forge-installer security sweep dispatched 2026-04-30T15:59:07Z).

### 2.3 `docs/faq.md` — frequently-asked questions

**Audience:** Anyone with a quick question that doesn't merit a full guide.
**Scope:** ~15-25 entries seeded from existing reviewer questions, channel discussions, and the obvious things first-time users ask. Includes: what's the difference between InterGenOS and other LFS-derived distros; what does "Security ONLY, not Security First" mean in practice; can I install proprietary software (yes, via `pkm install-helper`); what about telemetry (none); what about voice (text-only by design); how do I add my own kernel modules (DKMS + your own MOK).
**Dependency:** None — can be written today and grown organically post-launch.

### 2.4 `docs/hall-of-fame.md` — researcher acknowledgment page

**Audience:** Security researchers who responsibly disclose vulnerabilities to InterGenOS.
**Scope:** Currently referenced in `SECURITY.md` line 96 ("Hall of Fame: we will publish a researcher-acknowledgment page alongside the v1.0 launch") but does not exist yet. v1.0 launch needs this published with at least the framework even if no entries exist on day one. Placeholder is acceptable; the page itself is not optional.
**Dependency:** None — placeholder writable today.

### 2.5 `docs/shim-review-submission.md` — published submission materials

**Audience:** rhboot/shim-review reviewers, plus any future maintainer who wants to understand the submission posture.
**Scope:** The graduated form of `docs/research/shim_review/README_draft_skeleton.md`. Once the 39Q template is filled and the PR opens at `github.com/rhboot/shim-review` (target 2026-05-15), the submission content lives at a published path so reviewers and the public can read it without spelunking through `docs/research/`.
**Dependency:** Ceremony complete (PGP fingerprints + vendor cert), 39Q gated questions answerable (B2 Dockerfile artifacts, SBAT extraction, etc.), shim-review PR opened. Currently mid-flight; SPOC + WC own the path forward.

### 2.6 `docs/repository-trust.md` — pkm repository trust model

**Audience:** Any user trying to verify InterGenOS is what it claims to be — and anyone packaging InterGenOS for downstream redistribution.
**Scope:** How `pkm` verifies an archive (sha256 against repo index, signature chain to the release signing key, the `--archive-trust strict|loose|repo-only` flag introduced in pkm H3, what the keyring package contains and how it rotates). Cross-references `signing-key.md` (fingerprints) and `signing-procedure.md` (signing flow). Anchored on the principle that the update path itself is part of the attack surface.
**Dependency:** Ceremony complete (signing-key fingerprints) + pkm H3 lands (already on master at `c9d7462` per 4th-batch).

## 4. End-user documentation

These items target someone who has downloaded an InterGenOS image, installed it, and wants to use it. Several overlap with Section 2.

### 4.1 Getting started (`docs/getting-started.md`)

### 4.2 Install guide (`docs/install-guide.md`)

See Section 2.2. Forge installer walkthrough.

### 4.3 First-boot reference (`docs/first-boot-greeter.md`)

**Current:** On master at `7c03d58` (today). End-user-facing for the front sections, peer-developer-facing for Internals.
**v1.0 status:** Already shipping; no further action.

### 4.4 FAQ (`docs/faq.md`)

See Section 2.3.

### 4.5 InterGen user guide (`docs/intergen-user-guide.md`)

**Audience:** A user who knows the AI assistant exists and wants to actually use it.
**Scope:** What InterGen is, what it can do (per-tier capabilities), how to invoke it, the permission model (every tool call gated), the audit log, opt-in cloud providers (Glasswing-Anthropic, Gemini-Google, etc.), Local-Rules vs Local-Qwen Sentinel routing. Cross-references `intergen(1)`.
**v1.0 status:** Should ship at v1.0 with current scope (CLI + D-Bus surface, no GTK4 panel since panel is paused per drift audit DRIFT-6).

### 4.6 Sentinel user guide (`docs/intergen-sentinel.md`)

**Audience:** A user who wants to understand or configure the security-scanner layer in front of MCP tool calls.
**Scope:** What Sentinel is, what scanning happens, the local-default-only posture, how to opt-in to a cloud provider, what each provider sees (and does not see), how to swap/disable scanners, where the audit log lives. Connects to InterGen user guide and to the project's security-only alignment.
**v1.0 status:** Should ship at v1.0 with the local-only Path 1 + the cloud-provider opt-in surface; full Glasswing implementation is in stub state per DRIFT-7 and may need scope-truncation in this doc.

### 4.7 pkm common tasks (`docs/pkm-guide.md`)

**Audience:** Users who want a friendlier walkthrough than the man page.
**Scope:** Walking through the common workflows — install + remove + search + verify, install-helper for proprietary software, repo configuration, signature verification, what `--archive-trust` does, troubleshooting. Cross-references `pkm(1)`.
**v1.0 status:** Recommended for v1.0; useful even if minimal at launch.

### 4.8 Building from source (`docs/building-from-source.md`)

**Audience:** Power users + future maintainers who want to build their own InterGenOS image rather than installing the published one.
**Scope:** Build-host requirements (already partially covered in `CONTRIBUTING.md`), the `build-intergenos.sh` orchestrator phases, how to use checkpoints, what the build produces, how to validate it. Cross-references `CONTRIBUTING.md` and the `igos-build/` source.
**v1.0 status:** Recommended for v1.0; may already be partially covered in README. If README's coverage is enough, this consolidates rather than expands.

## 6. Gap analysis — what to write next

### 6.1 Load-bearing for v1.0 (write before ship)

| File | Effort | Reason |
|---|---|---|
| `docs/hall-of-fame.md` | <1 hour | Already promised in `SECURITY.md`; placeholder satisfies the promise. Skip and we ship with a broken cross-reference. |
| `docs/getting-started.md` | 2-3 hours | First thing every new user looks for. Skip and we ship with a confused first impression. |
| `docs/faq.md` | 2-3 hours | Frequently-asked-question pressure builds quickly post-launch; pre-seeding 15-25 entries from current channel discussions saves owner+SPOC time. |
| `docs/repository-trust.md` | 2-3 hours | Documents pkm trust model + ceremony output. Reviewer-facing AND user-facing; skip and we ship with an undocumented trust path. |
| `docs/intergen-user-guide.md` | 3-4 hours | InterGen is a flagship feature per README; users will hit it on day one. |
| `docs/intergen-sentinel.md` | 2-3 hours | Sentinel is the security story for InterGen; user guide is incomplete without it. Scope can shrink to "what's available at v1.0" given DRIFT-7. |
| `docs/install-guide.md` | 4-6 hours, partial today | Forge UI must be reachably stable before screenshots; prose can be drafted now from the design plan and updated post-stable. |
| `docs/shim-review-submission.md` | 1 hour, post-PR-open | Mechanical graduation of the 39Q skeleton once the PR opens (2026-05-15). |
| `docs/signing-key.md` (real fingerprints) | <30 min, post-ceremony | Fingerprint fill from today's ceremony output. |

**Total v1.0-must-ship effort:** ~17-25 hours of writing, much of it parallelizable. Most can be drafted now and refined as dependencies land.

### 6.2 Owner recommendations

1. **Today (post-ceremony):** Fill `docs/signing-key.md` with the real fingerprints. This is a <30-min mechanical fill that unblocks Section 6.1's repository-trust doc and the public statements that depend on the fingerprint being live.

2. **This week:** Write `docs/hall-of-fame.md` (placeholder), `docs/getting-started.md`, and `docs/faq.md`. These are the lowest-dependency, highest-pain-if-missing items. None require new dependencies; all three can be drafted from existing content.

3. **Pre-PR-open (before 2026-05-15):** Write `docs/intergen-user-guide.md`, `docs/intergen-sentinel.md`, and `docs/repository-trust.md`. These ship the v1.0 user-facing story and document the trust posture reviewers will probe.

4. **Post-ceremony, pre-launch:** Draft `docs/install-guide.md` from the design plan + existing `installer/tests/README.md` + the Forge SB integration work. Add screenshots once the TUI is presentation-stable.

5. **Day-of-launch:** Graduate `docs/research/shim_review/README_draft_skeleton.md` to `docs/shim-review-submission.md` as a mechanical move. The research/ copy can stay as the audit trail.

### 6.3 What this proposal does *not* recommend

- **No subsystem READMEs for v1.0.** `installer/`, `pkm/`, `igos-build/`, `intergen/` are readable from source and well-organized; a README per subsystem would be nice but not load-bearing. v1.0+1 candidate.
- **No per-package READMEs.** LFS/BLFS upstream documentation is sufficient; writing 654 of our own is scope creep no user has asked for.
- **No marketing content.** README does the marketing work. `docs/` is for substantive reference, not landing-page prose.
- **No "deep dive" architecture docs at v1.0.** `docs/research/` already holds the architecture audit trail. Curating a reader-friendly version is v1.0+1.
