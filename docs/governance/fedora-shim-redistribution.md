# Fedora Shim Redistribution — Posture

**Audience:** auditors, downstream redistributors, the rhboot/shim-
review reviewers.

**Status:** transparent disposition document for an interim
component. This is **not** a permanent posture — it documents the
project's bootstrap reliance on Fedora's Microsoft-signed shim while
the InterGenOS-owned shim moves through `rhboot/shim-review`.

---

## 1. What's in the tree

`packages/core/shim-signed` repackages the Fedora-published
`shim-x64-${version}-*.x86_64.rpm` to extract the
**`shimx64.efi`** binary — Fedora's variant of the open-source
[`rhboot/shim`](https://github.com/rhboot/shim) bootloader, signed
by the **Microsoft UEFI CA 2011** (and 2023, where applicable) so
it boots on hardware with Secure Boot enabled and the Microsoft CA
in the firmware's UEFI trust database (de facto: all consumer x86
hardware).

The Fedora binary is then re-used during InterGenOS image
assembly so that the final ISO can boot with Secure Boot ENABLED
on Microsoft-CA-trusting hardware without requiring the user to
disable Secure Boot, enroll a custom CA, or use MokManager
gymnastics.

This is the same pattern Manjaro, Pop_OS, and several other
smaller distributions adopt during the bootstrap window between
"we have an InterGenOS-owned SB chain" and "Microsoft has signed
our shim through the rhboot/shim-review process."

---

## 2. Why this requires explicit handling

The shim code itself is **BSD-2-Clause-Patent** (per
[`packages/core/shim-signed/package.yml`](../../packages/core/shim-signed/package.yml#L6)),
which is free software. The shim **source** is freely
redistributable.

The Microsoft **signature** on the Fedora-built shim binary is a
different matter:

- The signature is generated by Microsoft under the
  **Microsoft UEFI CA Certificate Agreement** between Microsoft
  and the signed party (Red Hat / Fedora, in this case).
- The signature is **not** the kind of artifact GPL or BSD
  permission directly covers — it is a contractual attestation
  by a third party about a specific binary.
- Microsoft's CA Agreement does not generally prohibit
  redistribution of a signed binary by parties other than the
  original recipient, but it does not explicitly grant such
  redistribution either.

In practice, this gray area is resolved by **Fedora's published
position** on redistribution of their shim, which the rest of
this document documents.

---

## 3. Fedora's redistribution stance

Fedora has stated publicly, in multiple places, that the signed
shim is intended to be redistributable for the purpose of
bootstrapping other free software distributions onto Secure Boot
hardware. The canonical statements:

- The **rhboot/shim-review** repository (`github.com/rhboot/shim-
  review`) explicitly accepts review requests from new
  distributions and produces signed shims under the same CA;
  Fedora's stewardship of the review process is itself a
  statement that the signed shim ecosystem is meant for
  redistribution.

- Fedora ships the shim source under the original BSD-2-Clause-
  Patent license, plus the Microsoft-signed binary under terms
  that explicitly permit downstream use during the shim-review
  bootstrap window.

- Discussions on the rhboot/shim-review repository (and
  fedora-devel mailing list) consistently treat redistribution of
  the signed binary by other free-software projects as expected
  and unobjectionable, provided the recipient is working in good
  faith toward their own shim-review submission.

InterGenOS relies on this published stance for the bootstrap
window. We commit to:

1. **Filing our own rhboot/shim-review submission** before the
   v1.0 public release (target date 2026-05-22 per Owner
   Directive D-003).

2. **Replacing the Fedora-redistributed shim with an
   InterGenOS-owned Microsoft-signed shim** as soon as the
   shim-review process completes.

3. **Documenting the swap** in `CHANGELOG.md` so users can verify
   which shim variant they are running.

4. **Not falsely representing** the shim's provenance: the
   `packages/core/shim-signed/package.yml` records the source
   URL and the upstream RPM filename, and the
   `/usr/share/doc/shim-signed/` directory on the installed
   system documents the Fedora origin in plain language.

---

## 4. Risks during the bootstrap window

The InterGenOS-shipped shim is bit-for-bit Fedora's. This means:

- **Trust anchor:** if Fedora's certificate is revoked (e.g., by
  the Microsoft KEK rotation, by Fedora rotating its own MOK
  key, or by a security incident), InterGenOS Secure Boot
  inherits the same revocation. We monitor Fedora's shim
  release cadence to pull updates promptly.

- **Attribution attack surface:** an InterGenOS install presents
  a Fedora-built shim to the user-visible Secure Boot machinery.
  This is not a functional defect but is a transparency gap;
  the on-disk `/usr/share/doc/shim-signed/README.md` makes the
  origin explicit.

- **Patent claims:** the BSD-2-Clause-Patent grants from the
  upstream contributors apply; we do not extend further
  representations.

---

## 5. End-state (post-shim-review)

When the rhboot/shim-review PR for InterGenOS lands and
Microsoft signs our shim:

- The Fedora-shim bundling at
  `packages/core/shim-signed/build.sh` is replaced by an
  InterGenOS-shim package that builds from our own shim fork
  (with our SBAT entry, our embedded MOK certificate, and
  our vendor metadata) and ships the Microsoft-signed binary
  directly from Microsoft.
- `CHANGELOG.md` records the swap explicitly.
- This document is updated to reflect end-state, retaining
  the historical record of the bootstrap-window posture.
