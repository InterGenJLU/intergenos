# Build System Survey — Modern Approaches Beyond Bash

**Date:** March 31, 2026
**Context:** InterGenOS build system design — researching alternatives to pure bash for LFS-style builds

---

## Systems Surveyed

### 1. ALFS / jhalfs (Automated Linux From Scratch)

- **Language:** Bash (primary), Python port in progress
- **How it works:** Downloads LFS book XML, extracts commands via XSLT, generates executable shell scripts
- **Repository:** https://github.com/automate-lfs/jhalfs
- **Status:** Actively maintained, rolling release
- **Key insight:** Python port exists, indicating community desire to move beyond bash

### 2. Buildroot

- **Language:** Make + Kconfig
- **Philosophy:** Simplistic, minimalistic — avoids complexity
- **Template system:** .mk (Makefile) + Config.in files per package
- **Package infrastructure:** Standardized rules per build system type (pkg-python.mk, pkg-cmake.mk)
- **BR2_EXTERNAL:** Mechanism for extending without modifying core
- **Output:** Full root filesystem image (monolithic — must regenerate for updates)
- **Best for:** Embedded/IoT, small teams, tight deadlines

### 3. Yocto / OpenEmbedded

- **Build engine:** BitBake (inspired by Gentoo Portage)
- **Language:** BitBake DSL with embedded Bash and Python
- **Recipe files:** .bb (recipes), .bbappend (overrides), .bbclass (classes), .conf (config), .inc (includes)
- **Workflow:** Parse metadata → generate dependency graph → execute tasks (fetch→unpack→patch→configure→compile→install→package)
- **Output:** Distribution with package feed (allows individual package updates)
- **Best for:** Ambitious multi-platform products, regulated industries

### 4. Gentoo Portage / ebuilds

- **Language:** Python (Portage) + Bash (ebuilds)
- **Architecture:** ebuild files define per-package build logic; Portage handles dependency resolution
- **eclasses:** ~200+ reusable function libraries (autotools, bash-completion, flag-o-matic, etc.)
- **USE flags:** Fine-grained feature selection at compile time
- **Slots:** Parallel installation of multiple versions
- **Key pattern:** User-facing templates in Bash, system-facing orchestration in Python
- **Best for:** Desktop/server customization, power users

### 5. Void Linux xbps-src

- **Language:** Bash framework
- **Repository:** https://github.com/void-linux/void-packages
- **Key innovation:** `build_style` abstraction (gnu-configure, cmake, meson, make, cargo, custom)
- **Build environment:** Chroot isolation (bwrap, xfail, chroot)
- **Cross-compilation:** First-class support for foreign architectures
- **Dependency types:** hostmakedepends, makedepends, checkdepends, depends
- **Best for:** Medium complexity packages, clean balance of abstraction and control

### 6. NixOS / Nix

- **Language:** Nix (custom purely functional, lazily-evaluated, dynamically-typed)
- **Philosophy:** Packages as pure functions; immutable after building
- **Flakes:** Standard structure with lock files for exact reproducibility
- **Key advantage:** Eliminates dependency hell via hash-based directory isolation
- **Key disadvantage:** Steep learning curve, opaque to traditional Linux users

### 7. CRUX Linux

- **Language:** Bash (Pkgfiles)
- **Philosophy:** Minimal abstraction, LFS-adjacent, pure transparency
- **Port structure:** Pkgfile + .footprint + .md5sum + patches
- **Dependency tracking:** Header comments only (# Depends on:)
- **Single build() function:** Developer manually sequences all phases
- **Best for:** Minimal systems, learning, LFS users

### 8. Arch Linux (PKGBUILD)

- **Language:** Bash
- **Functions:** prepare(), build(), check(), package()
- **Dependency types:** depends, makedepends, checkdepends, optdepends
- **Best for:** Simple to moderately complex packages, maximum control

### 9. Alpine Linux (APKBUILD)

- **Language:** Bash
- **Functions:** prepare(), build(), check(), package()
- **Cross-compilation:** makedepends_build vs makedepends_host
- **Subpackages:** Easy package splitting
- **Best for:** Lightweight systems, containers, minimal dependencies

## Comparative Analysis

### Abstraction Levels (Low → High)

1. CRUX (none — direct bash scripts)
2. Arch (minimal — pure bash functions)
3. Alpine (low — bash functions, subpackages)
4. Void (high — build_style abstractions)
5. Gentoo (medium — eclasses + phase functions)
6. Yocto (highest — classes, inheritance, Python)

### Feature Flags / Conditional Features

| System | Support | Mechanism |
|--------|---------|-----------|
| Gentoo | Powerful | USE flags with conditional dependencies |
| Yocto | Yes | PACKAGECONFIG, EXTRA_OECONF |
| All others | No | Manual conditionals only |

### Build-time vs Runtime Dependency Declaration

| System | Distinction | Slots | Version Specs |
|--------|-------------|-------|---------------|
| Gentoo | Yes (DEPEND/RDEPEND/BDEPEND) | Yes | Extensive |
| Yocto | Yes (DEPENDS/RDEPENDS) | No | Simple |
| Void | Implicit (makedepends/depends) | No | Simple |
| Alpine | Yes (makedepends/depends) | No | Simple |
| Arch | Implicit (makedepends/depends) | No | Simple |
| CRUX | No | No | No |

## Key Takeaways for InterGenOS

1. **Bash templating is proven-sufficient** for 90% of packages (CRUX, Void, Alpine success)
2. **Python orchestration adds value** once scale exceeds ~200 packages or cross-compilation needed
3. **Void's build_style pattern** is the best balance of abstraction and transparency
4. **Gentoo's hybrid model** (Python orchestrator + Bash templates) is proven at massive scale
5. **Dependency resolution doesn't need to be NP-complete** — topological sort works in practice
6. **Binary caching is critical** — nobody tolerates rebuilding GCC every time
7. **Modular template inheritance** scales better than monolithic definitions

## Decision

InterGenOS will use: **Python orchestrator + YAML package templates + Bash build functions**
- YAML for declarative metadata (readable, machine-parseable)
- Bash for imperative build steps (configure/make/install is shell work)
- Python for orchestration (dependency graphs, build ordering, validation, caching)
- Build styles inspired by Void's approach
