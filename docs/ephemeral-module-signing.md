# Ephemeral Per-Build Kernel Module Signing

**Last updated:** 2026-04-21

InterGenOS signs kernel modules with a per-build key that is generated at build time, embedded into the kernel image it's paired with, and discarded when the build completes. This document describes the posture, its properties, and why it is the right fit for a source-built distribution under Secure Boot.

It is the canonical reference for reviewers and users who want to verify how our module-signing trust chain works. The shim-review submission cites this document in its vendor-cert and key-management answers.

## Summary

| Property | Value |
|---|---|
| Signing key lifetime | One kernel build |
| Signing key storage | Never persisted; generated in `certs/signing_key.pem` at build time, removed at end of build |
| Public half | Embedded in the built kernel's `.builtin_trusted_keys` keyring |
| Scope | In-tree kernel modules only; never signs a bootloader, never signs out-of-tree modules |
| Out-of-tree modules (DKMS) | Signed by a per-machine Machine Owner Key (MOK) generated by the Forge installer at first boot; chained via `CONFIG_SECONDARY_TRUSTED_KEYRING=y` |
| Kernel config used | `CONFIG_MODULE_SIG=y`, `CONFIG_MODULE_SIG_ALL=y`, `CONFIG_MODULE_SIG_FORCE=y`, `CONFIG_SECONDARY_TRUSTED_KEYRING=y`, no `CONFIG_MODULE_SIG_KEY` set |
| Vendor cert interaction | **None.** The ephemeral key and the distro EFI vendor cert are distinct keys with distinct purposes. |

## How It Works

The upstream Linux kernel build with `CONFIG_MODULE_SIG=y` and no explicit `CONFIG_MODULE_SIG_KEY` setting performs the following sequence:

1. `make` auto-generates an RSA-4096 X.509 keypair at `certs/signing_key.pem`.
2. Kernel compilation compiles the public half of that keypair into `vmlinuz`'s `.builtin_trusted_keys` keyring.
3. Every in-tree loadable `.ko` file is signed with the private half of that keypair via `scripts/sign-file`.
4. The private half is not persisted. No file outside the build tree references it. When the build tree is cleaned (or discarded, as it is for release builds), the private half is gone.
5. Output artifacts: one `vmlinuz` image plus a set of signed modules, every one of which verifies against the pubkey embedded in that same `vmlinuz`. The image and its modules are a matched unit.

At boot: the kernel is loaded by our distro-signed GRUB (signed with the distro EFI vendor cert, which lives on the hardware token — a separate key). The kernel verifies each module it loads against its built-in keyring. Under `CONFIG_MODULE_SIG_FORCE=y` any module whose signature does not verify is rejected.

Because the signing keypair is discarded after each kernel build, **every kernel image has its own module-signing pubkey, paired only with its own signed modules.** Modules from kernel X cannot be loaded into kernel Y's runtime — by design.

## Why This Is the Right Fit

**Minimum persistent attack surface.** The compromise window for a module-signing key is the duration of a single kernel build. Stealing a persisted module-signing key would let an attacker sign arbitrary modules for every kernel that key signs. An ephemeral key narrows that to "signatures valid only for this one kernel image," which dramatically reduces the value of stealing the key — a stolen ephemeral key has no reuse value past its own build.

**Strong binding between modules and the kernel they target.** Because the kernel embeds the pubkey that signed its own modules, there is no possibility of a third-party key collision producing modules a given kernel would accept. Each (kernel, module-set) pair is cryptographically anchored.

**No persistent trust anchor to audit, rotate, back up, or lose.** Module signing does not add a long-lived operational burden to the distribution maintenance lifecycle. We have long-lived keys for the things that need long-lived continuity (distro GPG for release signing; EFI binary cert for the bootloader chain), and ephemeral keys for the things that don't (kernel module signing).

**Independent of hardware-token availability.** The hardware token with the distro EFI vendor cert is only needed at release-signing time (kernel image + GRUB `sbsign` step). Module signing happens inside the kernel build, on the build host, with no token interaction. This lets us split the build and sign steps cleanly: builds run in an igos-build VM, final signatures happen on the signing workstation with the token plugged in.

## User-Side DKMS / Out-of-Tree Modules

InterGenOS does not ship kernel modules outside the in-tree set. When a user opts in to a third-party kernel module (common example: the NVIDIA proprietary driver, if a user installs it via a download helper), the distro's ephemeral key plays no role. The flow instead is:

1. The Forge installer generates a Machine Owner Key (MOK) on the end user's machine at install time.
2. On first boot, the user confirms enrollment of their MOK in the firmware via MokManager.
3. DKMS builds the third-party module against the installed kernel headers and signs it with the user's MOK private half.
4. The kernel loads the module — `CONFIG_SECONDARY_TRUSTED_KEYRING=y` allows the MOK-enrolled keyring to chain into the module-verification path.

Key properties of this flow:

- **The distro does not hold any key capable of signing a user's out-of-tree modules.** The MOK is created on the user's machine and never leaves it.
- **The distro has no visibility into which out-of-tree modules a given user has installed or signed.** DKMS signing is entirely a user-side operation.
- **Compromise of the distro EFI vendor cert does not enable bypassing module signature enforcement on users' machines.** MOK-enrolled out-of-tree modules remain trustworthy even under that threat model.

## What The Ephemeral Key Does NOT Do

Explicitly, for reviewer clarity:

- **It does not sign a bootloader.** GRUB is signed by the distro EFI X.509 key on PIV slot 9c of the hardware token. That key is fully distinct from any module-signing key.
- **It does not sign a kernel image.** `vmlinuz` is signed by the same distro EFI X.509 key used for GRUB.
- **It does not sign modules from any previous or future kernel build.** Each kernel build has its own keypair; cross-build module acceptance does not occur.
- **It does not sign third-party / out-of-tree / DKMS modules.** Those are signed by per-user MOKs generated by the Forge installer.
- **It is not persisted to the hardware token, the build host filesystem, a repository, or any backup.** It exists for the duration of one kernel build.
- **It is not a "vendor cert" in the shim-review sense.** Our vendor cert is the distro EFI X.509 cert embedded in our custom shim; it is published, long-lived, and hardware-token-backed. The ephemeral module-signing key has no relationship to it.

## Relationship to the Three Distro Keys

InterGenOS manages three distinct keys for release and boot-chain integrity, with a fourth (the MOK) owned entirely by end users. Detail table:

| # | Key | Purpose | Lifetime | Custody |
|---|---|---|---|---|
| 1 | Distro GPG | Signs `pkm` repository index; signs release artifacts | 2 years / 5 years (root) | Hardware token, offline root, paper + LUKS USB backup |
| 2 | Distro EFI X.509 | Signs `vmlinuz` + `grubx64.efi` via `sbsign` | 2 years / tied to root | Hardware token (PIV slot 9c) |
| 3 | **Kernel module-signing X.509** | **Signs in-tree kernel modules** | **One kernel build** | **Generated in-build, discarded in-build** |
| 4 | MOK (Machine Owner Key) | Signs DKMS / out-of-tree modules on each end user's machine | Per-machine, user-owned | End user only — never the distro |

This document is about key #3. Keys #1 and #2 are documented in [signing-key.md](signing-key.md) and [signing-procedure.md](signing-procedure.md). Key #4 is user-owned; its generation is handled by Forge's MOK setup flow (`installer/backend/mok.py` in the repo).

## Independent Verification

A reviewer can confirm this architecture end-to-end without trusting our documentation. The commands below run against a published InterGenOS kernel package.

### Verify the kernel embeds its own module-signing pubkey

```
# Per D-016, scratch artifacts live under ~/tmp/<workflow>/, not /tmp.
mkdir -p ~/tmp/kernel-builtin-keys-inspect

# Extract the signing pubkey embedded in the kernel image.
extract-vmlinux /boot/vmlinuz-<version>-intergenos > ~/tmp/kernel-builtin-keys-inspect/vmlinux.elf
objcopy --dump-section .builtin_trusted_keys=~/tmp/kernel-builtin-keys-inspect/builtin.keys ~/tmp/kernel-builtin-keys-inspect/vmlinux.elf

# Dump the embedded keyring.
openssl x509 -in ~/tmp/kernel-builtin-keys-inspect/builtin.keys -noout -text -inform DER
```

### Verify a shipped module was signed by that pubkey

```
# Show the signature block of a kernel module.
modinfo -F signer /lib/modules/<version>-intergenos/kernel/fs/ext4/ext4.ko
modinfo -F sig_key /lib/modules/<version>-intergenos/kernel/fs/ext4/ext4.ko

# These should match the CN and fingerprint of the pubkey embedded
# in the kernel image above.
```

### Verify enforcement at runtime

```
cat /proc/sys/kernel/module_sig_enforce
# 1

sysctl kernel.modules_disabled
# 0

cat /sys/kernel/security/lockdown
# [integrity] or [confidentiality]
```

Every `.ko` loaded on a running InterGenOS system will verify under these settings. An unsigned module, or one signed by any other key, fails to load and emits a kernel-log message.
