# v1.1 Deferred Follow-ups

This document is the canonical ledger of v1.0 / v1.1 scope splits. Every commit that
references future-version-deferral framing (e.g. *"deferred to v1.1"*, *"post-v1.0"*,
*"v1.1 follow-up"*) must either:

- Add a new entry here (in the same commit set), OR
- Reference an existing entry by ID (F-NNN format) in the commit body.

The pre-push gate at [`.githooks/pre-push`](../.githooks/pre-push) Gate 9 enforces
this consistency, so a scope deferral cannot quietly land without being registered.

Re-engagement of any entry requires explicit operator authorization per D-009 item 5.
The trigger field on each entry names the concrete condition that should re-open the
work; nothing here re-opens on time alone.

## F-001 — Filesystem snapshots for upgrade rollback

**Source**: T0-5 design question 1 (Rollback / snapshot model), operator greenlit 2026-05-19.

**Scope deferred to v1.1**: btrfs subvolume snapshots and ZFS snapshots as transactional rollback substrates for `pkm upgrade`. The v1.0 implementation ships per-package archive cache plus install-new-first ordering, which is filesystem-agnostic and works on the ext4 installs Forge currently produces. Filesystem-level snapshots are the strictly-better long-term rollback mechanism — single atomic revert, file metadata preservation, near-zero overhead on copy-on-write filesystems — but they're blocked on the Forge installer offering btrfs as a supported filesystem option.

**Operational reason for the v1.0 / v1.1 split**: The Forge installer currently commits to ext4 only. Adding btrfs as a filesystem option requires its own design pass — installer UX, GRUB configuration for btrfs subvolumes, initramfs subvolume mount support, default subvolume layout, snapshot retention policy. Doing both filesystem-level snapshots in pkm and btrfs in Forge simultaneously would have stalled the upgrade-safety closure work behind installer work that has nothing to do with package management correctness. The per-package archive cache fallback is correct on its own merits and remains the default path even after btrfs lands.

**Re-engagement trigger**: btrfs is added as a Forge installer-supported filesystem option. When that lands, fold filesystem-level snapshot integration into pkm's upgrade orchestration as an opt-in fast path. The per-package archive cache stays as the default and the fallback for ext4 systems — this is additive scope, not a rewrite.

**Status**: OPEN — v1.0 ships the per-package-cache + install-new-first path at SC5 closure.

## F-003 — Reverse-dep warning is the only v1.0 cascade-safety mechanism

**Source**: T0-5 audit row O-033 Phase 1 (Cascade-rebuild needed on library upgrade is invisible), operator-greenlit 2026-05-19. Rescoped at sprint closure 2026-05-19 from earlier framing that named Phase 2 SONAME-aware refuse semantics as v1.1 work; the project is not currently committed to delivering any SONAME-aware mechanism.

**v1.0 scope** (the only mechanism shipped + the only mechanism the project commits to):

- `pkm.cascade.format_reverse_dep_warning` (landed at `da304855`) renders a human-readable warning at upgrade time: "upgrading <name> from X to Y will affect N reverse dependencies." Reads the existing `depends` table populated via the `add_depends` infrastructure from H-004 / `dbdb9533`.
- The warning is informational; the user retains control over whether to proceed with the upgrade.

This works because the user already has the data they need (which packages reverse-depend on the upgrade target). What it does NOT do is automatically distinguish SONAME-preserving bumps (libfoo 1.2.3 → 1.2.4, same SONAME) from SONAME-breaking bumps (libfoo 1.2.3 → 2.0.0, SONAME changed, every dependent must rebuild) — that distinction requires SONAME ground-truth in the index, which v1.0 does not capture.

**Explicitly not a project commitment** (any of these would be separate operator-authorized scope, not a roadmap item):

- Per-library SONAME extraction at package-build time (ELF header parsing OR `package.yml` schema declaration).
- A `SONAME` column on the `files` table (or a sibling `library_metadata` table).
- Refuse-on-SONAME-bump semantics with `--allow-soname-break` override.
- Auto-include of SONAME-affected reverse-dependents in the same transactional upgrade batch.

The Phase 1 helper landed in v1.0 means a natural integration surface exists for any future SONAME-aware mechanism (`format_reverse_dep_warning` would become the cosmetic-only path while a new `check_soname_break_required` would return a hard-refuse signal) — but no such mechanism is committed.

**Re-engagement trigger**: explicit operator authorization to scope a SONAME-tracking design pass. There is no time-based or empirical-signal-based trigger; the ledger entry survives as the audit-trail record of which mechanisms v1.0 ships vs which would require a separate scope-grant.

**Status**: RATIFIED 2026-05-19 — v1.0 ships the Phase 1 reverse-dep warning helper; alternate SONAME-aware mechanisms are non-committed scope.

## F-002 — Hand-curated-only security-advisory authoring

**Source**: T0-5 design question 7 (Security-only upgrade filtering), operator-greenlit 2026-05-19. Rescoped at sprint closure 2026-05-19 from earlier framing that had named automated upstream-feed ingestion as v1.1-deferred work; the project is not currently committed to delivering any alternate-input mechanism.

**v1.0 scope** (the only mechanism shipped + the only mechanism the project commits to):

- `docs/governance/security-advisories.yml` is hand-edited by InterGenOS maintainers as they patch CVEs.
- `scripts/generate-repodb.py` reads the file at index-build time and stamps `security: true` + `cves: [string]` on matching package versions in `InterGenOS.db`.
- `pkm upgrade --security-only` is a boolean filter: candidates with `security=true` only.

The path is unglamorous but accurate — InterGenOS knows which packages it patched for which CVEs because it patched them.

**Explicitly not a project commitment** (any of these would be separate operator-authorized scope, not a roadmap item):

- Ingestion of upstream advisory sources (NVD, OSV, Debian Security Tracker, distro-specific feeds).
- Per-package severity scoring derived from upstream metadata.
- Severity-band filtering (`--security-only=critical,high`).
- A `pkm changelog --security` view of recent security-flagged entries.

The schema fields are forward-compatible with future inputs, but no future input is committed — absence of a commitment is itself the disposition.

**Re-engagement trigger**: explicit operator authorization to scope an alternate-input design pass. There is no time-based or milestone-based trigger; the ledger entry survives as the audit-trail record of which mechanisms v1.0 ships vs which would require a separate scope-grant.

**Status**: RATIFIED 2026-05-19 — v1.0 ships hand-curated; alternate mechanisms are non-committed scope.

## F-005 — write_file path-aware risk-tier classification

**Source**: T0-4-E SPOC design-level peer-review clarification C1 at 2026-05-19T23:26:20Z. SPOC flagged that RFC §6 example differentiates write_file by target path (write_file to $HOME = USER_SCOPE; write_file to /etc/ = PRIVILEGED) but IGOSC's v1.0 `_classify_risk_tier()` treats write_file unconditionally as PRIVILEGED_STATE_CHANGING via the `_PRIVILEGED_TOOLS` frozenset. SPOC's verbatim observation: "Either is acceptable for v1.0; if conservative default, document as v1.x backlog item."

**v1.0 scope** (the only mechanism shipped): write_file is in `intergen/tool_registry._PRIVILEGED_TOOLS` frozenset unconditionally. Every write_file invocation routes through the pkexec runner per Step 15 contract. This over-prompts $HOME writes (the user authenticates to write a file under their own home directory) but is the safer default — pickup-failure costs at most extra prompts; mis-classification costs unauthenticated /etc/ writes.

**Scope held for v1.x** (operator-authorization to engage required per D-009 item 5):

- Extend `_classify_risk_tier(tool, arguments, tool_name)` to inspect `arguments['path']` for `write_file` calls.
- USER_SCOPE_STATE_CHANGING when path resolves under the calling user's `$HOME` (excluding `$HOME/.ssh/`, `$HOME/.gnupg/`, `$HOME/.config/autostart/`, `$HOME/.local/share/applications/` — security-sensitive subdirs stay PRIVILEGED to compose with PROTECTED_PATHS).
- PRIVILEGED_STATE_CHANGING when path resolves under `/etc/`, `/usr/`, `/var/`, `/boot/`, `/root/`, `/srv/`, or any path outside the calling user's HOME.
- Path resolution uses `Path(arg_path).expanduser().resolve()` to defeat traversal tricks; symlink targets are followed (the gate cares about the actual file's location, not the symlink's location).
- Add per-tool ToolSchema.classify_risk_tier() optional override hook so tools that need path-aware logic can declare it themselves (extensibility for future fine-grained tools without registry edits).

**Operational reason for the v1.0 / v1.x split**: path-aware classification adds three new code paths to the v1.0 gate (path resolution, classification override hook, symlink resolution semantics) at a point in the cycle where the gate's correctness invariants need to be peer-reviewed in isolation. v1.0 ships the conservative wide net so the dispatch path semantics are uncluttered; v1.x narrows once the gate is in users' hands and the UX cost of over-prompting is quantified by telemetry.

**Re-engagement trigger**: telemetry from v1.0 shows over-prompting on $HOME writes is degrading UX OR a new privileged-path category needs fine-grained policy (e.g., writes under `/var/lib/<my-app>/` for a specific app that owns its data directory).

**Status**: OPEN — v1.0 ships conservative unconditional PRIVILEGED_STATE_CHANGING for write_file.

## F-006 — ToolSchema.is_ingress declaration replaces hardcoded INGRESS_TOOLS_V1 frozenset

**Source**: T0-4-E SPOC design-level peer-review clarification C4 at 2026-05-19T23:26:20Z. SPOC flagged extensibility: "How does IngressTracker know which tools count? Per-tool declaration on ToolSchema (e.g. is_ingress=True) vs hardcoded list vs decorator? Extensibility matters for v1.x when new tools may need ingress classification — confirming the mechanism supports declaration alongside the tool, not requiring an IngressTracker edit per new tool."

**v1.0 scope** (the only mechanism shipped): `intergen/interfaces/provenance.INGRESS_TOOLS_V1` is a hardcoded frozenset of the 5 known ingress tools at v1.0 surface: `read_file` + `read_url` + `web_search` + `read_clipboard` + `list_directory`. `IngressTracker.__init__(ingress_set=INGRESS_TOOLS_V1)` accepts the frozenset by parameter (already extensible by injection if a future caller wants a different set), but the canonical set is hardcoded.

**Scope held for v1.x** (operator-authorization to engage required per D-009 item 5):

- Add `is_ingress: bool = False` field to `intergen/interfaces/types.ToolSchema` dataclass.
- Mark each ingress tool's schema with `is_ingress=True` at the tool definition site (alongside `safety_tier`, `description`, etc. — the per-tool metadata that already lives with the tool).
- Replace the hardcoded `INGRESS_TOOLS_V1` frozenset with a derived set computed at tool-registry discovery time: `{tool.name for tool in registry if tool.schema.is_ingress}`.
- Document the schema field in the tool-authoring guide so new ingress tools declare themselves rather than needing an `IngressTracker` source edit.
- Preserve `INGRESS_TOOLS_V1` as a fallback frozenset for external-handler tools (MCP / cloud) whose schema is set by the external source and may not carry the `is_ingress` field.

**Operational reason for the v1.0 / v1.x split**: the v1.0 ingress set is small (5 tools) and known. The cost of an extensibility refactor at v1.0 ship time would be schema migration across every tool + a new ToolRegistry derivation path + composition checks with the existing per-tool safety classification — work that the v1.0 gate's correctness validation does not need. v1.x absorbs this once the gate's invariants are battle-tested and new ingress tools are actively being authored.

**Re-engagement trigger**: a new ingress tool is added to intergen (or an existing tool acquires an ingress-shaped behavior) that would otherwise require an `IngressTracker` source-level edit to recognize.

**Status**: OPEN — v1.0 ships hardcoded `INGRESS_TOOLS_V1` frozenset; extensibility lives behind `IngressTracker.__init__(ingress_set=...)` parameter injection for callers that want a custom set.

## Adding new entries

When operator authorization splits scope between v1.0 and v1.1:

1. Pick the next available F-NNN ID from the table above.
2. Add a row to the Index table with title, source, re-engagement trigger, status.
3. Add the full entry below following the F-001 / F-002 shape: source, scope deferred, operational reason, re-engagement trigger, status.
4. Reference the F-NNN ID in the commit body of the v1.0 scaffold commit. The pre-push Gate 9 will validate the reference.

When a re-engagement trigger fires:

1. Confirm with the operator before starting the v1.1 work — D-009 item 5 prohibits self-directed re-engagement.
2. Update the entry's Status field with the v1.1 implementation work and link to the closing commit(s).
3. Do not delete the entry once closed — it stays as historical record of the scope-split decision.

## Why this exists

Without an enforced ledger, "land the scaffold now, do the real work in v1.1" becomes the shape of every quietly-lost task — the scaffold convinces the team the work is done, the real work is forgotten, the user inherits a half-built feature with no record of what's missing. The ledger plus the pre-push gate make this failure mode impossible by construction: every deferral is registered, every deferral has an explicit re-engagement trigger, and the gate refuses commits that try to skip the bookkeeping.
