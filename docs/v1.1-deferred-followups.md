# v1.1 Deferred Follow-ups

This document is the canonical ledger of v1.0 / v1.1 scope splits. Every commit that
references future-version-deferral framing (e.g. *"deferred to v1.1"*, *"post-v1.0"*,
*"v1.1 follow-up"*) must either:

- Add a new entry here (in the same commit set), OR
- Reference an existing entry by ID (F-NNN format) in the commit body.

The pre-push gate at [`.githooks/pre-push`](../.githooks/pre-push) Gate 9 enforces
this consistency, so a scope deferral cannot quietly land without being registered.

Re-engagement of any entry requires explicit operator authorization per D-009 item 5.
The trigger field on each entry names the concrete condition that should re-open the
work; nothing here re-opens on time alone.

## F-001 — Filesystem snapshots for upgrade rollback

**Source**: T0-5 design question 1 (Rollback / snapshot model), operator greenlit 2026-05-19.

**Scope deferred to v1.1**: btrfs subvolume snapshots and ZFS snapshots as transactional rollback substrates for `pkm upgrade`. The v1.0 implementation ships per-package archive cache plus install-new-first ordering, which is filesystem-agnostic and works on the ext4 installs Forge currently produces. Filesystem-level snapshots are the strictly-better long-term rollback mechanism — single atomic revert, file metadata preservation, near-zero overhead on copy-on-write filesystems — but they're blocked on the Forge installer offering btrfs as a supported filesystem option.

**Operational reason for the v1.0 / v1.1 split**: The Forge installer currently commits to ext4 only. Adding btrfs as a filesystem option requires its own design pass — installer UX, GRUB configuration for btrfs subvolumes, initramfs subvolume mount support, default subvolume layout, snapshot retention policy. Doing both filesystem-level snapshots in pkm and btrfs in Forge simultaneously would have stalled the upgrade-safety closure work behind installer work that has nothing to do with package management correctness. The per-package archive cache fallback is correct on its own merits and remains the default path even after btrfs lands.

**Re-engagement trigger**: btrfs is added as a Forge installer-supported filesystem option. When that lands, fold filesystem-level snapshot integration into pkm's upgrade orchestration as an opt-in fast path. The per-package archive cache stays as the default and the fallback for ext4 systems — this is additive scope, not a rewrite.

**Status**: OPEN — v1.0 ships the per-package-cache + install-new-first path at SC5 closure.

## F-003 — SONAME tracking + refuse-upgrade on SONAME bump without --allow-soname-break

**Source**: T0-5 audit row O-033 Phase 2 (SONAME cascade-rebuild needed on library upgrade), operator greenlit 2026-05-19.

**Scope deferred to v1.1**: Per-library SONAME extraction at package-build time, persistence in the `files` table (or a sibling `library_metadata` table) as a SONAME column, and upgrade-time refuse semantics that block an upgrade when `old.SONAME != new.SONAME` for any library shipping reverse-dependents — except when the user passes `--allow-soname-break` to acknowledge the cascading rebuild work, or when the upgrade orchestration auto-includes the SONAME-affected reverse-dependents in the same transactional batch.

**Operational reason for the v1.0 / v1.1 split**: The Phase 1 v1.0 implementation (informational reverse-dependency warning at upgrade time, via `pkm.cascade.format_reverse_dep_warning`) gives the user the data needed to decide whether to proceed with a library upgrade. It does not block on SONAME divergence because we lack SONAME ground-truth in the index — without that, we cannot tell the difference between a security-patch bump (libfoo 1.2.3 → 1.2.4, same SONAME, no rebuild needed) and a major-version bump (libfoo 1.2.3 → 2.0.0, SONAME changed, every dependent must rebuild). Adding SONAME tracking requires either: (a) parsing the build output's ELF headers at package-build time to extract SONAME, or (b) declaring SONAME in the `package.yml` schema and trusting maintainers to keep it accurate. Both are real design surfaces.

**Re-engagement trigger**: Post-v1.0-ship. Once the v1.0 user base is stable and we have empirical data on how often library upgrades cause user-visible breakage from Phase 1's warning-only posture, design the SONAME-extraction pipeline as a v1.1 project. The Phase 1 warning helper landed in v1.0 means there's a natural integration surface — `format_reverse_dep_warning` becomes the cosmetic-only path while a new `check_soname_break_required` returns a hard-refuse signal.

**Status**: OPEN — v1.0 ships the Phase 1 reverse-dep warning helper. Phase 2 SONAME tracking deferred.

## F-002 — Automated CVE feed ingestion + severity scoring + `pkm changelog --security`

**Source**: T0-5 design question 7 (Security-only upgrade filtering), operator greenlit 2026-05-19.

**Scope deferred to v1.1**: Automated ingestion of upstream CVE feeds (NVD, OSV, Debian Security Tracker, distribution-specific feeds), per-package severity scoring derived from upstream metadata, severity-band filtering on `pkm upgrade --security-only=critical,high`, and a `pkm changelog --security` view that shows recent security-flagged entries with CVE references for review before upgrade.

**Operational reason for the v1.0 / v1.1 split**: Automated feed integration requires a CVE-feed-to-package-version mapping pipeline that is itself a substantial design surface: which upstream feeds are authoritative for which packages; how to match upstream-CVE-product-version-ranges against our version numbers when our patching diverges from upstream; how to reconcile disagreements between feeds; how to handle the latency between disclosure and feed publication; how to keep the pipeline running without each new CVE feed becoming a per-package-maintainer chore. That work belongs to its own design pass. v1.0 ships the schema fields (`security: bool` and `cves: [string]` per-version-entry in `InterGenOS.db`) plus a hand-curated `security-advisories.yml` in the InterGenOS source tree, populated as we patch CVEs. This is unglamorous but accurate — we know which packages we've patched for which CVEs because we patched them.

**Re-engagement trigger**: Post-v1.0-ship. Once the v1.0 user base and index format have stabilized, design the CVE-feed-to-package-version mapping pipeline as a v1.1 project. The schema fields landing in v1.0 mean indexes do not need to be rewritten or re-signed retroactively when automation arrives — automation just populates fields that already exist.

**Status**: OPEN — v1.0 ships the schema fields plus hand-curated path at SC5 closure.

## Adding new entries

When operator authorization splits scope between v1.0 and v1.1:

1. Pick the next available F-NNN ID from the table above.
2. Add a row to the Index table with title, source, re-engagement trigger, status.
3. Add the full entry below following the F-001 / F-002 shape: source, scope deferred, operational reason, re-engagement trigger, status.
4. Reference the F-NNN ID in the commit body of the v1.0 scaffold commit. The pre-push Gate 9 will validate the reference.

When a re-engagement trigger fires:

1. Confirm with the operator before starting the v1.1 work — D-009 item 5 prohibits self-directed re-engagement.
2. Update the entry's Status field with the v1.1 implementation work and link to the closing commit(s).
3. Do not delete the entry once closed — it stays as historical record of the scope-split decision.

## Why this exists

Without an enforced ledger, "land the scaffold now, do the real work in v1.1" becomes the shape of every quietly-lost task — the scaffold convinces the team the work is done, the real work is forgotten, the user inherits a half-built feature with no record of what's missing. The ledger plus the pre-push gate make this failure mode impossible by construction: every deferral is registered, every deferral has an explicit re-engagement trigger, and the gate refuses commits that try to skip the bookkeeping.
